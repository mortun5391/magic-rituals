# Roadmap реализации Early Game для Magic Rituals

## Цель документа

Этот документ описывает не лор и не общий дизайн, а именно порядок реализации раннего этапа мода.

Задача roadmap:

- разбить большой early game на маленькие этапы
- определить, что делать сначала, а что позже
- не утонуть в объёме
- сохранить цельность лора, исследования, ритуалов и ресурсов
- дать практические подсказки по внедрению каждой системы

Документ опирается на уже существующее состояние проекта:

- есть базовая система ритуалов
- есть блоки рун
- есть поэтапное поглощение предметов
- есть два ритуала: `Raincaller` и `Clear Skies`
- есть частицы, звуки, отмена и менеджер активных ритуалов

Значит теперь задача — превратить это в настоящий ранний этап прогрессии, а не в два отдельных ритуала.

## Главный принцип реализации

Не надо пытаться делать весь early game сразу.

Правильный порядок:

1. зафиксировать фундамент данных
2. сделать минимальный рабочий цикл исследования
3. связать его с книгой
4. только потом расширять контентом, структурами и существами

Если начать с большого количества ресурсов, биомов, мобов и красивого UI без базового цикла исследования, мод быстро разъедется на набор полуготовых систем.

## Что является MVP раннего этапа

Ниже минимальный рабочий ранний цикл, после которого мод уже начнёт ощущаться цельным:

1. `Пыль Пелены`
2. `Осколок Эха`
3. `Путеводитель по Пелене`
4. `Игла Следа`
5. `Сырой Отпечаток Течения`
6. `Чаша Осаждения`
7. `Осаждённая Эссенция Течения`
8. `Плита Разметки`
9. `Ритуальный мел`
10. `Стол Дешифровки`
11. первая открываемая глава знания
12. требование знания для `Raincaller`
13. вторая исследовательская ветка через Жар
14. требование знания для `Clear Skies`

Если этот цикл работает, всё остальное можно уже наращивать сверху.

## Общий порядок реализации

Лучше разбить работу на 8 этапов:

1. `Фундамент данных`
2. `Базовые ресурсы и предметы`
3. `Механика следов и Игла Следа`
4. `Обработка следов`
5. `Книга и knowledge`
6. `Требование знания для ритуалов`
7. `Worldgen и точки интереса`
8. `Полировка, баланс и расширение`

Ниже всё это расписано подробно.

## Этап 0. Подготовка архитектуры

### Цель

Перед добавлением контента нужно подготовить структуру кода, чтобы новые системы не развалили текущую ритуальную основу.

### Что сделать

Добавить новые логические пакеты:

- `knowledge`
- `research`
- `item`
- `world` или `worldgen`
- `block.entity` если пойдёшь через block entities

Пример структуры:

```text
com.mortun.magicrituals
  block
  item
  knowledge
  research
  rituals
  registry
  world
```

### Подсказка

Не держи систему знаний внутри `rituals`.  
Ритуалы должны только спрашивать:

- знает ли игрок это знание
- может ли игрок активировать данный ритуал

Сама система знаний должна жить отдельно.

### Что не надо делать на этом этапе

- не начинать с UI книги
- не добавлять сразу 20 ресурсов
- не делать позднегеймовые исследования

## Этап 1. Фундамент данных

### Цель

Сначала нужно определить, что именно будет храниться как знание, след и результат исследования.

### Что нужно ввести

#### 1. Типы знаний

Минимально:

- `FIRST_TRACE`
- `FRAGILE_ECHO`
- `TRACE_NEEDLE`
- `TRACE_OF_FLOW`
- `SETTLING`
- `RITUAL_CHALK`
- `LISTENING_PATTERN`
- `RAINCALLER`
- `STORM_MARKS`
- `TRACE_OF_HEAT`
- `ASHEN_RESONANCE`
- `CLEAR_SKIES`

Это могут быть:

- enum
- набор `ResourceLocation`
- реестр кастомных knowledge id

### Рекомендация

Для начала лучше взять простой enum-like подход через `ResourceLocation`, а не жёсткий Java enum.  
Так потом будет проще выносить знания в data-driven формат.

#### 2. Типы следов

Минимально:

- `FLOW`
- `HEAT`
- `GROWTH`

Позже:

- `ECHO`
- `STONE`
- `BREACH`

#### 3. Стадии следа

Нужно различать:

- источник в мире
- сырой отпечаток
- обработанный фрагмент
- знание, полученное из фрагментов

То есть логика должна быть такой:

```text
world trace source -> raw trace sample -> processed resonance fragment -> unlocked knowledge
```

### Как это реализовывать

Заведи отдельные сущности данных:

- `KnowledgeId`
- `TraceType`
- `TraceSampleData`
- `ResearchEntryId`

### Детализация этапа 1: что создать и где

Ниже — минимальный набор сущностей и их назначение. Это не геймплей, а чисто фундамент данных.

#### 1. Идентификатор знания

**Файл/пакет:** `com.mortun.magicrituals.knowledge`  
**Сущность:** `KnowledgeId`

**Зачем:** единый ключ для всего знания (глав, ритуалов, прогресса игрока).  
**Как:** простой wrapper над `ResourceLocation` или тип alias.

**Минимальный функционал:**

- хранит `ResourceLocation`
- сравнение и `toString()`
- фабрика `KnowledgeId.of(String)`

#### 2. Типы следов

**Файл/пакет:** `com.mortun.magicrituals.research`  
**Сущность:** `TraceType`

**Зачем:** различать базовые направления (FLOW/HEAT/GROWTH), чтобы дальше из них строились знания.  
**Как:** `enum` или id-реестр.

**Минимальный функционал:**

- набор базовых типов
- преобразование в id (например, `magicrituals:flow`)

#### 3. Стадия следа

**Файл/пакет:** `com.mortun.magicrituals.research`  
**Сущность:** `TraceStage`

**Зачем:** формализовать путь: источник -> сырой отпечаток -> обработанный фрагмент -> знание.  
**Как:** `enum` или sealed-like класс.

**Минимальный функционал:**

- `SOURCE`
- `RAW_SAMPLE`
- `PROCESSED_FRAGMENT`
- `KNOWLEDGE`

#### 4. Данные отпечатка

**Файл/пакет:** `com.mortun.magicrituals.research`  
**Сущность:** `TraceSampleData`

**Зачем:** переносить “след” через предметы и блоки обработки.  
**Как:** простой DTO/record.

**Минимальный функционал:**

- `TraceType type`
- `TraceStage stage`
- `ResourceLocation sourceId` (если нужно связать с источником)
- сериализация в NBT / data component

#### 5. Идентификатор research-записи

**Файл/пакет:** `com.mortun.magicrituals.research`  
**Сущность:** `ResearchEntryId`

**Зачем:** связать конкретный путь исследования с главой книги.  
**Как:** аналогично `KnowledgeId` — wrapper над `ResourceLocation`.

**Минимальный функционал:**

- хранит `ResourceLocation`
- фабрика `ResearchEntryId.of(String)`

#### 6. Хранилище знания игрока (минимум)

**Файл/пакет:** `com.mortun.magicrituals.knowledge`  
**Сущность:** `PlayerKnowledge`

**Зачем:** фиксировать, что игрок уже знает.  
**Как:** Set `KnowledgeId` + методы доступа.

**Минимальный функционал:**

- `boolean has(KnowledgeId id)`
- `void grant(KnowledgeId id)`
- `Set<KnowledgeId> getAll()`

#### 7. Реестр знания (минимум)

**Файл/пакет:** `com.mortun.magicrituals.knowledge`  
**Сущность:** `KnowledgeRegistry`

**Зачем:** единый список доступных знаний + быстрый поиск по id.  
**Как:** статическая карта или registry-like класс.

**Минимальный функционал:**

- `register(KnowledgeId id, KnowledgeDefinition def)`
- `Optional<KnowledgeDefinition> get(KnowledgeId id)`

#### 8. Определение знания (минимум)

**Файл/пакет:** `com.mortun.magicrituals.knowledge`  
**Сущность:** `KnowledgeDefinition`

**Зачем:** описывать, что именно это за знание.  
**Как:** record/POJO.

**Минимальный функционал:**

- `KnowledgeId id`
- `String titleKey`
- `String descriptionKey`
- `ResearchEntryId unlocksFrom` (опционально)

---

#### Итого для этапа 1

В коде должно появиться:

- `knowledge`: `KnowledgeId`, `KnowledgeDefinition`, `KnowledgeRegistry`, `PlayerKnowledge`
- `research`: `TraceType`, `TraceStage`, `TraceSampleData`, `ResearchEntryId`

И этого достаточно, чтобы дальше строить предметы, блоки обработки и книгу уже на понятном фундаменте.

### Практический совет

Если на старте не хочешь вводить сложные data components, можно хранить тип следа через:

- `ItemStack` + data component / custom data
- или разные предметы на каждый тип

Для MVP проще:

- отдельный предмет `raw_flow_trace`
- отдельный предмет `settled_flow_essence`

А уже потом переходить к более универсальному формату.

### Итог этапа

На выходе у тебя должны быть чётко названы:

- знания
- типы следов
- типы ранних исследовательских предметов

Без этого дальше лучше не идти.

## Этап 2. Базовые ресурсы и предметы

### Цель

Добавить первые новые предметы и материалы, из которых складывается early game.

### Что внедрять первым

#### Предметы сбора

- `Пыль Пелены`
- `Осколок Эха`
- `Следовая Смола`
- `Меловой Сланец` или его предметная форма

#### Предметы исследования

- `Путеводитель по Пелене`
- `Игла Следа`
- `Сырой Отпечаток Течения`
- `Сырой Отпечаток Жара`
- `Осаждённая Эссенция Течения`
- `Осаждённая Эссенция Жара`

#### Предметы практики

- `Основа мела`
- `Ритуальный мел`
- `Золотой ритуальный мел` если оставляешь текущую схему

### Как внедрять

Порядок:

1. добавить предметы в `ModItems`
2. добавить модели и локализацию
3. добавить временные рецепты или заглушки
4. только потом начинать привязывать логику

### Подсказка

На старте не трать время на сложные крафты.  
Сначала обеспечь полный игровой цикл, потом уже усложняй рецепты.

### Пример микроэтапов

#### Этап 2.1

Добавить только:

- Пыль Пелены
- Осколок Эха
- Иглу Следа
- Путеводитель

#### Этап 2.2

Добавить:

- Сырой Отпечаток Течения
- Осаждённую Эссенцию Течения

#### Этап 2.3

Добавить:

- Меловой Сланец
- Основу мела
- Ритуальный мел

### Что тестировать

- все предметы отображаются
- все рецепты собираются
- ничего не ломает текущую ритуальную систему

## Этап 3. Система знания игрока

### Цель

Сделать основу knowledge, без которой книга и доступ к ритуалам будут фиктивными.

### Что должно уметь знание

Система должна:

- храниться на игроке
- переживать выход из мира
- синхронизироваться на клиент
- выдавать знание
- проверять наличие знания

### Как лучше реализовать

Оптимальный путь:

- player attachment / capability-подобное хранилище
- отдельный класс вроде `PlayerKnowledge`

Пример API:

```java
boolean hasKnowledge(ServerPlayer player, ResourceLocation knowledgeId)
void grantKnowledge(ServerPlayer player, ResourceLocation knowledgeId)
Set<ResourceLocation> getKnowledge(ServerPlayer player)
```

### Что делать первым

Сначала вообще без книги:

1. создать storage знаний
2. дать тестовую команду или временный debug-предмет
3. убедиться, что знание выдаётся и сохраняется

### Подсказка

Это один из самых важных этапов.  
Если knowledge storage будет плохим, потом переделывать придётся всё:

- книгу
- исследования
- блокировки ритуалов
- открытие записей

### Что тестировать

- знание выдаётся
- знание не теряется после relog
- знание можно проверять из кода ритуала

## Этап 4. Игла Следа и поиск источников

### Цель

Сделать первую “живую” механику исследования в мире.

Это не просто предмет, а фактическая точка входа в магический gameplay.

## Подэтап 4.1. MVP Иглы

### Что должно работать

- игрок держит Иглу в руке
- рядом с валидным источником следа предмет реагирует
- игрок может снять отпечаток

### Как сделать максимально просто

Сначала не нужен сложный алгоритм поиска.

Можно сделать так:

- в мире есть конкретные блоки-источники или метки-источники
- Игла проверяет вокруг игрока радиус, например 6-8 блоков
- если находит валидный источник — начинает сигналить

### Как игрок взаимодействует

Минимальная схема:

1. Игрок берёт Иглу в руку.
2. Подходит к валидному месту.
3. Игла начинает светиться.
4. Игрок кликает ПКМ по блоку-источнику.
5. Получает сырой отпечаток.

### Почему так лучше начать

Потому что это дешёвый MVP.  
Не нужно сразу делать:

- сложную наводку по направлению
- длительное удержание
- мини-игру считывания

Сначала проверь, что сам цикл работает.

## Подэтап 4.2. Нормальная версия Иглы

После MVP можно улучшать.

### Что добавить

- изменение звука по расстоянию
- частицы около источника
- пульсацию предмета в руке
- короткое действие считывания 1-2 секунды
- возможность провала, если игрок сдвинулся

### Как это выглядит в коде

Лучше держать логику отдельно:

- `TraceNeedleItem`
- `TraceLocator`
- `TraceSourceResolver`

Пример ответственности:

- `TraceNeedleItem`: реакция на ПКМ, клиентские/серверные сигналы
- `TraceLocator`: поиск ближайшего валидного источника
- `TraceSourceResolver`: знает, какие блоки/условия считаются следом

### Подсказка

Не зашивай все условия прямо в item-класс.  
Иначе позже сломаешься на добавлении новых типов следов.

## Подэтап 4.3. Источники следов

Нужно решить, что именно в раннем MVP будет источником следа.

Лучший путь для начала:

- не делать абстрактные “невидимые зоны”
- использовать понятные блоки и структуры

Примеры ранних источников:

- особый налёт Пыли Пелены возле воды
- кристалл Эха возле аметиста
- жаровой блок-маркер рядом с магмой

Позже можно добавить:

- невидимые аномалии
- динамические погодные следы
- временные узлы после бурь

### Что тестировать

- Игла реагирует только на валидные источники
- Игла не спамит эффектами постоянно
- снятие отпечатка не дюпает ресурс

## Этап 5. Обработка следов

### Цель

Сделать так, чтобы сырой отпечаток сам по себе не был конечным предметом.

Это важнейший этап, потому что именно он создаёт глубину и отличает исследование от обычного сбора лута.

## Подэтап 5.1. Чаша Осаждения

### MVP

Сначала можно сделать очень простую механику:

- блок с GUI
- 2-3 слота
- один рецепт на Течение

Пример:

- слот 1: `Сырой Отпечаток Течения`
- слот 2: вода или предмет-катализатор
- слот 3: стабилизатор
- выход: `Осаждённая Эссенция Течения`

### Как реализовывать

Лучше как отдельный блок + block entity + menu.

Классы:

- `SettlingBasinBlock`
- `SettlingBasinBlockEntity`
- `SettlingBasinMenu`
- `SettlingRecipe` или простой ручной процесс на старте

### Подсказка

Для первого варианта можешь вообще не делать полноценный кастомный recipe type.  
Если рецептов мало, допустим временный ручной маппинг в block entity.

Но если сразу видишь, что рецептов будет много, лучше делать recipe system с самого начала.

## Подэтап 5.2. Плита Разметки

### MVP

Сначала:

- перерабатывает меловой сланец в основу
- из основы + Пыли Пелены делает мел

### Как внедрять

Аналогично:

- блок
- block entity
- menu

Можно делать позже, чем Чашу, но до полной книги.

### Практический совет

Если хочешь сократить объём, сначала можно временно оставить мел на верстаке, а Плиту добавить уже после рабочего исследования Течения.

Но идеальный early game лучше строится именно через отдельный блок.

### Что тестировать

- переработка идёт
- нет дюпов
- рецепты легко расширяются

## Этап 6. Книга и главы

### Цель

Сделать книгу не просто предметом, а интерфейсом прогрессии игрока.

## Подэтап 6.1. MVP книги

На этом этапе не нужен полный Thaumcraft-style UI.

Минимум:

- предмет-книга
- экран с одной колонкой глав
- открытые и закрытые записи
- правая панель текста

### Что книга должна уметь

- читать знания игрока
- показывать открытые главы
- показывать закрытые главы как скрытые
- отображать намёки

### Как это реализовывать

Сделай data model отдельно от UI:

- `GuideEntry`
- `GuideEntryId`
- `GuideRegistry`

Пример полей у записи:

- id
- title
- hint
- body
- requiredKnowledge
- icon
- unlockedRecipes
- unlockedRituals

### Подсказка

Не храни текст глав хардкодом в экран-классе.  
Иначе потом обновление книги станет мучением.

Лучше:

- сначала Java registry
- позже возможно перейти на json/datapack-подобный формат

## Подэтап 6.2. Привязка глав к знаниям

Книга не должна сама “открывать” знание.  
Она только отображает уже открытое.

Логика должна быть такой:

- исследование выдало знание
- знание сохранилось у игрока
- книга увидела знание и открыла запись

### Что тестировать

- запись открывается только после выдачи знания
- скрытая запись показывает намёк
- ритуал по-прежнему заблокирован без знания

---

## Этап 7. Стол Дешифровки и настоящая механика исследования

### Цель

Сделать центральную механику понимания следов.

Это важнее, чем просто “получить дроп с мира”, потому что именно тут наблюдение превращается в знание.

## Подэтап 7.1. MVP дешифровки

Сделай сначала очень простой вариант.

Например, для открытия главы `Зов Дождя` нужно:

- 2 `Осаждённая Эссенция Течения`
- 1 `Меловой Якорь`

Игрок кладёт всё это на стол и получает знание `RAINCALLER`.

### Как это реализовать

Блок:

- `DecipheringTableBlock`
- `DecipheringTableBlockEntity`
- `DecipheringTableMenu`

Система:

- `ResearchPattern`
- `ResearchManager`

На старте pattern может быть очень простой:

- список обязательных входов
- выдаваемое знание

### Подсказка

Не строй сразу слишком сложную мини-игру.

Сначала важно проверить сам цикл:

- нашёл след
- обработал
- вставил на стол
- получил знание

## Подэтап 7.2. Улучшенная дешифровка

После MVP уже можно добавить:

- центральную схему, которая “загорается”
- частичное совпадение
- разные типы фрагментов
- логические группы

Пример:

- один фрагмент даёт 1 из 3 сегментов схемы
- правильный второй фрагмент зажигает ещё 2
- неверный третий вызывает рассогласование

### Как показывать игроку, что он близок к разгадке

Это очень важно.

Игрок должен получать промежуточный фидбек:

- слабое свечение при частично верной комбинации
- более яркое свечение при почти верной
- звук ошибки при конфликте
- всплывающую подсказку или цветовую реакцию узора

Так исследование ощущается как процесс понимания, а не как слепой подбор.

## Подэтап 7.3. Масштабирование сложности

Сразу при проектировании заложи 3 уровня сложности:

### Уровень A

- 2-3 входа
- один тип следа
- одна глава

### Уровень B

- 3-4 входа
- два типа следов
- нужен катализатор

### Уровень C

- часть данных надо добывать в мире
- часть ставить на стол
- нужен предварительный полевой шаг

Это важно, потому что поздний гейм должен расти из той же системы, а не из новой с нуля.

## Этап 8. Блокировка ритуалов знаниями

### Цель

Сделать так, чтобы знания были не декоративными.

### Что менять

Сейчас ритуал определяется через:

- структуру
- рецепт

Нужно добавить ещё:

- `requiredKnowledge`

### Как лучше встроить

В `Ritual` добавить метод:

```java
ResourceLocation requiredKnowledge();
```

Или:

```java
Optional<ResourceLocation> requiredKnowledge();
```

Затем в `ChalkRuneBlock` или `RitualManager` добавить проверку:

- игрок знает нужное знание?

Если нет:

- ритуал не стартует
- игрок получает красное сообщение
- в лог уходит причина
- проигрывается эффект незнания

### Подсказка

Эта проверка должна быть до запуска `RitualSession`.

То есть порядок такой:

1. золотая руна
2. структура совпала
3. рецепт совпал
4. знание открыто
5. только тогда стартует сессия

### Что тестировать

- ритуал не запускается без знания
- глава открылась -> ритуал запускается
- рецепты и структура сами по себе не обходят progression

## Этап 9. Связка книги и исследований с практикой

### Цель

Сделать так, чтобы исследование не заканчивалось просто открытием текста.

### Как должно быть

Некоторые знания должны иметь две стадии:

- `открыто`
- `подтверждено практикой`

Пример:

- игрок открыл `RAINCALLER`
- но глава `STORM_MARKS` не откроется, пока он реально не проведёт этот ритуал

### Как это реализовать

Есть два варианта:

#### Вариант 1

Просто отдельные knowledge id:

- `KNOW_RAINCALLER`
- `CAST_RAINCALLER`

#### Вариант 2

У знания есть stage:

- DISCOVERED
- UNDERSTOOD
- PRACTICED

### Рекомендация

Для early game проще вариант 1.  
Отдельные knowledge-флаги быстрее реализуются и лучше читаются при отладке.

## Этап 10. Worldgen и точки интереса

### Цель

Сделать так, чтобы исследования реально были частью мира.

## Подэтап 10.1. MVP worldgen

Сначала не нужны новые биомы.

Достаточно:

- маленьких feature-скоплений Пыли Пелены
- простых месторождений Мелового Сланца
- жаровых участков
- кристаллических включений Эха

### Почему это лучший старт

Потому что это:

- проще технически
- быстрее балансировать
- уже делает исследование географическим

## Подэтап 10.2. Малые структуры

После MVP добавить:

- Размытые Каменные Кольца
- Жаровые Выходы
- Мшистые Узлы

### Как внедрять

Порядок:

1. сначала feature-ресурсы
2. потом простая структура у воды
3. потом жаровая структура
4. потом живая структура

### Подсказка

Не вводи сразу много структур.  
Одна хорошо работающая структура важнее трёх сыроватых.

## Этап 11. Существа раннего этапа

### Цель

Усилить атмосферу и подать магию через живые отклики мира.

### Что делать первым

Если хочется мобов, лучше делать их позднее, чем исследовательский цикл.

Приоритет:

1. сначала следы
2. потом блоки обработки
3. потом книга
4. потом существа

### Почему

Потому что существа усиливают атмосферу, но сами по себе не формируют прогрессию.

### Лучший первый кандидат

`Следосвет`

Почему:

- не нужен сложный AI
- легко привязать к источникам следа
- хорошо работает как атмосферный указатель

### MVP-поведение

- появляется у некоторых узлов
- летает рядом
- не атакует
- иногда оставляет Пыль Пелены

### Что не делать сразу

- сложные боевые механики
- уникальные дроп-цепочки на 10 предметов
- обязательный моб-файт для ранней игры

## Этап 12. Подробный план внедрения предмета исследования

Ниже отдельный подробный mini-roadmap именно по `Игле Следа`.

## Шаг A. Добавить предмет как обычный item

Сначала:

- регистрируешь предмет
- даёшь модель
- добавляешь рецепт

Пока без логики.

## Шаг B. Добавить поиск ближайшего источника

Минимум:

- при тике в инвентаре или при использовании ищется валидный источник в радиусе

Лучше начать с:

- поиска только по server-side use
- а клиентские эффекты уже потом

## Шаг C. Добавить простое считывание

На ПКМ по источнику:

- проверка типа источника
- выдача сырого отпечатка
- визуальный и звуковой эффект

## Шаг D. Добавить реакцию предмета в руке

Когда рядом есть след:

- предмет слегка светится
- идёт звук или частицы

Это уже следующий слой, не первый.

## Шаг E. Добавить условия

Позже:

- время суток
- биом
- погода
- наличие нужной главы

### Подсказка по архитектуре

Не делай так:

```java
if (river && night && rain && ...
```

внутри одного item-класса на 200 строк.

Лучше:

- `TraceSource`
- `TraceCondition`
- `TraceCaptureResult`

То есть источник сам знает:

- можно ли с него считывать
- какой тип следа он даёт
- какие условия нужны

## Что можно смело отложить

Чтобы early game не затянулся в бесконечную разработку, можно отложить:

- сложные полевые конструкции исследования
- несколько видов мела
- сложную мини-игру дешифровки
- продвинутые стадии знания
- несколько новых существ
- новые биомы
- полноценные позднегеймовые эссенции

Сначала нужен рабочий стержень.

---

## Главные риски реализации

### 1. Слишком ранняя сложность

Если уже на первом исследовании игрок должен:

- искать редкий биом
- дождаться нужной погоды
- собрать 5 предметов
- разгадать сложную схему

то early game станет душным.

### 2. Слишком слабая связь между знанием и игрой

Если книга просто открывает текст, а механикам всё равно — знание станет бесполезным.

### 3. Слишком много предметов без функций

Если добавить 20 новых предметов до того, как они встроены в цикл — мод будет ощущаться мусорным и несобранным.

### 4. Слишком ранний UI-overengineering

Красивый интерфейс без готовой логики исследования замедлит разработку.

---

## Лучшая стратегия реализации

Самая разумная стратегия:

- сначала строить `вертикальный срез`
- потом расширять в ширину

Что такое вертикальный срез здесь:

1. игрок нашёл Пыль Пелены
2. сделал Иглу
3. снял след Течения
4. осадил его
5. открыл главу
6. провёл Raincaller

Вот это нужно сделать полностью раньше, чем:

- ветку Роста
- сложные структуры
- нескольких мобов
- продвинутый UI

Если вертикальный срез хорош, остальной early game уже строится намного увереннее.
